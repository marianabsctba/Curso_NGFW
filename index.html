CONCEITO HA
O que é High Availability (HA) em NGFW?
HA (Alta Disponibilidade) garante que o firewall continue operando mesmo se um nó falhar. Em NGFWs enterprise, existem dois modos principais:

• Active/Passive (A/P): apenas um nó processa tráfego; o standby assume em falha (failover ~1-3s)
• Active/Active (A/A): ambos os nós podem processar tráfego, com ganho de capacidade dependente do desenho da topologia, simetria de tráfego, licenciamento e modelo do fabricante

Este simulador representa um cluster Active/Active com sincronização de estado. Sessões TCP/UDP, tabela NAT e configuração podem ser sincronizadas por enlace dedicado de HA, permitindo failover rápido e preservação de sessões quando a sincronização de estado estiver habilitada, o tráfego for simétrico e a topologia for compatível.
// Protocolo HA simulado: heartbeat + sincronização de estado
Heartbeat interval: configurável | Dead interval: ajustado conforme topologia e fabricante
Session sync: TCP full state + UDP tracking + NAT table
Config sync: real-time push via TLS 1.3 AES-256-GCM
Failover trigger: perda de heartbeat, falha de interface monitorada, degradação de nó ou política de saúde do cluster

SD-WAN

POLICY-BASED
CONCEITO SD-WAN
O que é SD-WAN e por que é fundamental em NGFWs modernos?
SD-WAN (Software-Defined Wide Area Network) permite que o NGFW tome decisões inteligentes de roteamento de tráfego baseadas em métricas de qualidade de link em tempo real: latência, jitter, perda de pacotes e disponibilidade.

Em vez de simplesmente encaminhar tráfego pela rota default, o SD-WAN seleciona o melhor caminho para cada aplicação/serviço. Isso é especialmente crítico para:
• VoIP / Videoconferência: requer baixa latência e jitter
• SaaS (Microsoft 365, Salesforce): precisa de menor RTT possível
• Backup/Replicação: pode usar link de menor custo (failback)

O SD-WAN do Mariana BS integra-se com o controle de políticas do NGFW, então decisões de SD-WAN respeitam as regras de segurança e inspeção DPI.
// Algoritmos de seleção SD-WAN disponíveis:
1. Lowest Cost: escolhe link com menor custo configurado (manual)
2. Best Quality: baseado em SLA probe (ICMP/HTTP) — mede latência+jitter+pkt-loss
3. Bandwidth Load Balance: distribui sessões proporcionalmente à capacidade
4. Manual: admin escolhe o link por regra de política explícita
5. Application-Aware: DPI identifica aplicação → aplica SLA target específico

CONCEITO IPS/IDS
Intrusion Prevention vs Detection System
IDS (Intrusion Detection System) monitora passivamente o tráfego e gera alertas quando detecta padrões suspeitos, mas NÃO bloqueia.

IPS (Intrusion Prevention System) opera inline (em linha com o tráfego) e pode bloquear, dropar ou resetar conexões maliciosas em tempo real. É a evolução do IDS.

O Mariana BS NGFW opera em modo IPS puro: todas as inspeções são realizadas pelo motor ASIC/NPU antes do pacote ser encaminhado, com latência adicionada de apenas ~0.3ms.

Técnicas de detecção usadas:
• Signature-based: base de ~45.000 assinaturas (atualização horária)
• Anomaly-based: baseline de tráfego normal + desvio estatístico
• Protocol Decode: RFC compliance + validação de estados de protocolo
• Heurística/correlação/ML quando disponível: apoio à detecção comportamental, sempre combinado com telemetria e regras de segurança

SSL/TLS INSPECTION

DEEP INSPECT
CONCEITO SSL INSPECT
Como funciona a Inspeção SSL/TLS no NGFW?
Mais de 95% do tráfego web atual é criptografado via TLS. Sem inspeção SSL, o NGFW é "cego" para ameaças ocultas em HTTPS.

A inspeção SSL funciona como um Man-in-the-Middle controlado e autorizado:
1. O NGFW intercepta a conexão TLS do cliente
2. Cria uma nova sessão TLS até o servidor destino
3. Descriptografa, inspeciona o conteúdo (IPS, AV, DLP, etc.)
4. Re-criptografa e encaminha ao cliente

O firewall apresenta ao cliente um certificado re-assinado pela CA interna corporativa (que deve ser distribuída via GPO/MDM nos endpoints).

Protocolos observados/inspecionáveis conforme política: TLS 1.2 e TLS 1.3. TLS 1.0/1.1 devem ser tratados como legado e preferencialmente bloqueados em ambientes modernos.
Observação técnica: ECH/ESNI pode reduzir a visibilidade do SNI e exigir política de bypass, bloqueio ou controle por destino/categoria.
Cipher Suites bloqueadas: RC4, DES, 3DES, MD5, export ciphers, SWEET32

IPSec VPN

SITE-TO-SITE
CONCEITO IPSec
IPSec: Como funciona a criptografia de túneis VPN
IPSec é um conjunto de protocolos que provê autenticação e criptografia ao nível IP. Opera em dois modos:

• Transport Mode: criptografa apenas o payload; mantém cabeçalho IP original (host-to-host)
• Tunnel Mode: encapsula o pacote IP inteiro num novo pacote (site-to-site VPN)

O estabelecimento do túnel usa IKE (Internet Key Exchange) em duas fases:
Fase 1 (IKE SA): negocia algoritmos e troca chaves via DH (Diffie-Hellman). Cria um canal seguro para a Fase 2.
Fase 2 (IPSec SA): negocia os parâmetros reais do túnel (ESP/AH, algoritmos, SPI).

ESP (Encapsulating Security Payload): provê confidencialidade + autenticidade
AH (Authentication Header): apenas autenticidade (sem criptografia) — raramente usado hoje

Algoritmos recomendados 2026: IKEv2 + AES-256-GCM + SHA-384 + DH Group 21 (P-521)
// Parâmetros de Phase 1 (ISAKMP):
Encryption: AES-256-GCM | Hash: SHA-384 | DH: Group 21
Auth: RSA Certificate (ECDSA P-256) | Lifetime: 28800s
// Parâmetros de Phase 2 (ESP):
Transform: ESP-AES-256-GCM | PFS: DH Group 21 | Lifetime: 3600s / 4GB

Como funcionam as políticas de segurança em NGFWs?
As Security Policies (ou Access Control List — ACL) são o coração do firewall. Cada pacote que atravessa o NGFW é avaliado sequencialmente pelas regras (top-down, first-match).

Em NGFWs modernos, as políticas são multi-dimensional, podendo incluir:
• Zona de origem/destino, IP de origem/destino, Porta/Protocolo
• Usuário/Grupo (integração LDAP/AD), Aplicação (via DPI), URL Category
• Device Profile (tipo de dispositivo), Security Posture (certificado de saúde do endpoint)

O NGFW Mariana BS suporta políticas ZTNA (Zero Trust Network Access): cada acesso é verificado continuamente, não apenas na autenticação inicial.

CONCEITO ROUTING
Roteamento Dinâmico no NGFW: BGP e OSPF
NGFWs enterprise suportam protocolos de roteamento dinâmico, tornando-os participantes ativos da topologia de rede.

OSPF (Open Shortest Path First — IGP): protocolo de estado de enlace usado na rede interna. Mapeia toda a topologia e calcula rotas via algoritmo SPF (Dijkstra). Ideal para LAN corporativa e interconexão de sites.

BGP (Border Gateway Protocol — EGP): protocolo de vetor de path usado para peering com ISPs e na internet. No contexto do NGFW, é usado para:
• Receber rotas dos ISPs (eBGP)
• Anunciar prefixos próprios (PI address space)
• MPLS VPN com carrier (eBGP)
• SD-WAN overlay routing (iBGP)

Rota Default + Failover: via track de objeto (IP SLA) — se WAN1 cair, a rota default muda para WAN2 automaticamente.

NAT / PAT

STATEFUL
CONCEITO NAT
Network Address Translation: SNAT, DNAT, PAT e NAT64
NAT (Network Address Translation) permite que múltiplos hosts internos compartilhem um ou poucos IPs públicos.

• SNAT (Source NAT): traduz o IP de origem — usado para acesso à internet de hosts internos
• DNAT (Destination NAT): traduz o IP de destino — usado para publicar servidores internos na internet (Port Forwarding)
• PAT (Port Address Translation): variante do SNAT onde também a porta é traduzida — permite milhares de hosts com 1 IP público (overload)
• NAT64: traduz IPv6 para IPv4 e vice-versa
• Hairpin NAT: clientes internos acessando servidores DMZ pelo IP público

DLP: Prevenção de Fuga de Dados Sensíveis
DLP inspeciona o conteúdo do tráfego de saída em busca de dados sensíveis como CPF, CNPJ, cartões de crédito, dados LGPD, propriedade intelectual e PII.

Técnicas de detecção:
• Regex Patterns: padrões como CPF (\d{3}\.\d{3}\.\d{3}-\d{2}), PAN (cartão de crédito), etc.
• Fingerprinting: hash de documentos confidenciais catalogados — detecta cópias mesmo alteradas
• ML Classification: modelo treinado identifica dados sensíveis sem padrão fixo
• Keyword + Context: palavras como "Confidencial", "Internal Only", "Proprietary"

Canais monitorados: HTTPS (com SSL inspection), SMTP, FTP, HTTP upload, cloud drives.

Análise de Ameaças Desconhecidas por Detonação
O Sandbox recebe arquivos e URLs suspeitos e os executa em um ambiente isolado e controlado (VM/container), observando o comportamento para identificar malware de dia-zero que escapa de assinaturas tradicionais.

APPLICATION CONTROL

CONCEITO APP CONTROL
Controle de Aplicações via Deep Packet Inspection (DPI)
O App Control usa DPI para identificar a aplicação real independente da porta TCP/UDP usada. Isso é crítico porque aplicações modernas usam porta 80/443 (HTTP/HTTPS) para tudo.

Técnicas de identificação:
• Protocol Signature: padrões na camada 7 (ex: QUIC header de Google Meet)
• Behavioral Heuristics: características de fluxo (tamanho de pacote, timing, padrão de bytes)
• SSL SNI: hostname no handshake TLS (sem descriptografar)
• DNS Correlation: associa domínios com apps conhecidas
• ML Classification: rede neural classifica fluxos desconhecidos

QoS / TRAFFIC SHAPING

QoS garante que aplicações críticas recebam a largura de banda necessária mesmo em congestionamento.

Mecanismos do Mariana BS NGFW:

CTI / THREAT INTELLIGENCE

INTEL-DRIVEN SECURITY
CONCEITO CTI
O que é CTI dentro de um NGFW?
CTI (Cyber Threat Intelligence) é o uso de inteligência sobre ameaças para melhorar decisões de segurança. Em um NGFW, CTI normalmente alimenta reputação de IP, domínio, URL, arquivo, campanha, família de malware, infraestrutura de C2 e indicadores de comprometimento.

O ponto importante: CTI não é “lista mágica de bloqueio”. CTI boa precisa de contexto, confiança, validade temporal e ação proporcional. Um indicador pode ser malicioso hoje e irrelevante amanhã; por isso o NGFW deve registrar fonte, categoria, severidade, confiança e tempo de expiração do indicador.

Como o NGFW usa CTI:
• IP/domain reputation: bloqueio ou alerta para destinos conhecidos por malware, phishing, botnet ou C2
• URL intelligence: classificação por categoria, risco e reputação
• File reputation: consulta de hash e veredicto antes/depois do download
• IPS/AV enrichment: eventos ganham contexto de campanha, técnica e família de ameaça
• SIEM/SOC: logs enriquecidos ajudam investigação, correlação e priorização
// Boas práticas de CTI operacional:
Indicadores devem ter fonte, timestamp, confiança, severidade, tipo, TTL e contexto.
Bloqueio automático deve considerar risco, falso positivo e criticidade do ambiente.
CTI tática ajuda bloqueio rápido; CTI estratégica ajuda priorização e decisão executiva.
Integrações comuns: SIEM, SOAR, EDR/XDR, DNS Security, WAF, NDR e TIP.
